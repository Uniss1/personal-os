# Чеклист аудита LLM-проекта: 12 правил

Статусы:
- ✅ Выполнено
- ⚠️ Частично
- ❌ Не выполнено
- ⬜ Неприменимо
- ❓ Недостаточно данных

Для каждого правила фиксируй:
- Статус
- Доказательство
- Риск
- Следующий шаг

---

# Блок A. Основа проекта

## Правило 1. Четыре этапа LLM-проекта
- [ ] Есть четкая бизнес-постановка
- [ ] Зафиксированы целевые метрики
- [ ] Прототип сделан до начала полноценной разработки
- [ ] Прототип одобрен заказчиком по качеству
- [ ] Разработка опирается на прототип, а не на догадки
- [ ] Деплой включает постоянный мониторинг качества
- [ ] Учтен distribution shift

## Правило 2. Метрики качества
- [ ] Определен класс задачи:
  - точный ответ
  - проверяемый ответ
  - открытый ответ с продуктовыми критериями
- [ ] Есть оффлайн-метрики для быстрых итераций
- [ ] Продуктовые критерии качества записаны
- [ ] Есть способ оценки: асессоры или LLM-as-a-judge
- [ ] Если judge — есть чистый набор оценок
- [ ] Если люди — продумано качество разметки

## Правило 4. Прототип
- [ ] Есть выделенная команда
- [ ] Есть тестовая выборка
- [ ] Для тестовой выборки зафиксированы правильные ответы или критерии
- [ ] Метрики считаются на тестовой выборке
- [ ] Использовалась сильная доступная модель
- [ ] Итерации шли по схеме: промпт -> метрики -> анализ -> улучшение
- [ ] Прототип создан до production-разработки

## Правило 12. Итеративный подход
- [ ] Не строили "космолет" заранее
- [ ] Есть корзина реальных входных задач
- [ ] Корзина собирается из продукта, интервью или синтетики
- [ ] Качество оценивается систематически
- [ ] Проблемы кластеризуются по темам и причинам
- [ ] Гипотезы проверяются по одной
- [ ] Работает цикл: гипотеза -> реализация -> метрика -> решение

# Блок B. Архитектура

## Правило 3. Пять инструментов
- [ ] Сначала использован промптинг
- [ ] Промпт уточнялся через точные формулировки
- [ ] Проверялись few-shot / reasoning / structured output
- [ ] Большая задача при необходимости разбита на подзадачи
- [ ] RAG используется для знаний
- [ ] Fine-tuning рассматривается для поведения, а не знаний
- [ ] Tools подключены для действий вне текста
- [ ] Агентность применяется только если нельзя надежно задать алгоритм

## Правило 5. Контекст LLM
- [ ] В контекст добавлены нужные знания
- [ ] В контексте нет лишнего шума
- [ ] Размер контекстного окна контролируется
- [ ] Если заранее неясно, что подложить — используется retrieval
- [ ] В контексте учтены правила / история / tool outputs / состояние системы

### RAG-проверки
- [ ] Определен тип поиска: текстовый / векторный / структурированный / гибридный
- [ ] Поиск возвращает релевантные результаты
- [ ] Есть fallback, если поиск ничего не нашел
- [ ] Найденный текст корректно вставляется в контекст
- [ ] Понятно, где ломается система: в retrieval или generation

## Правило 6. Инструменты
- [ ] Детерминированная логика вынесена в код/инструменты
- [ ] LLM не подменяет калькулятор, сортировку, строгие проверки
- [ ] Structured output используется для аргументов и схем
- [ ] Инструменты описаны: вызов, аргументы, возврат
- [ ] Сначала пробовали обучить tool use через prompt
- [ ] Fine-tuning на tool use рассматривается только если prompt не справился

## Правило 7. Workflow vs Агенты
- [ ] По умолчанию используется workflow
- [ ] Этапы workflow понятны и управляемы
- [ ] На каждом этапе можно мерить качество отдельно
- [ ] Агент используется только при плохой формализуемости
- [ ] Результат агента верифицируем
- [ ] Есть понимание цены и надежности агентного подхода
- [ ] Число интентов / сценариев в workflow не разрастается бесконтрольно

# Блок C. Production и масштабирование

## Правило 8. Проверка ответов
- [ ] Есть critic-механизм
- [ ] Используется reflection и/или LLM-as-a-judge
- [ ] Понятно, когда ответ считать плохим
- [ ] Есть human-in-the-loop хотя бы для критичных случаев
- [ ] Если используется LLM-цикл — ограничено число итераций
- [ ] Кейсы, ушедшие на человека, собираются для улучшений

## Правило 9. Размер модели
- [ ] Отдельно оценены знания, рассуждения и следование промпту
- [ ] Знания компенсируются RAG там, где это возможно
- [ ] Рассуждающая большая модель используется только где это реально нужно
- [ ] Для рутинных задач рассмотрены малые модели
- [ ] Малая модель протестирована на тестовой выборке

## Правило 10. Дообучение
- [ ] Fine-tuning рассматривается только после prompt/RAG/workflow
- [ ] Fine-tuning не используется как загрузка знаний
- [ ] Есть качественный датасет
- [ ] Есть специалист, который умеет обучать модели
- [ ] Проверен риск усиления галлюцинаций
- [ ] После обучения метрики пересчитаны на тестовой выборке

## Правило 11. Оптимизации инференса
- [ ] Рассмотрен batching
- [ ] Рассмотрена quantization
- [ ] Настроен cache и/или KV-cache
- [ ] Рассмотрен speculative decoding
- [ ] Рассмотрены Flash / Paged Attention
- [ ] Выбран serving framework или осознанно зафиксировано, что serving контролирует внешний провайдер
- [ ] Ясен компромисс throughput vs latency
