# LLM Project Audit

Используй этот skill, когда нужно оценить существующий проект.

## Цель аудита
Не просто проставить галочки,
а понять:
- что уже работает
- где реальные риски
- что исправлять первым

---

## Обязательный порядок работы

### Шаг 1. Собери минимальный контекст

Запроси:
- бизнес-задачу
- пользователя
- целевые метрики
- текущую архитектуру
- используемые модели
- есть ли RAG
- есть ли tools
- workflow или agent
- как меряется качество
- есть ли тестовая выборка
- какие проблемы уже известны
- ограничения по бюджету / latency / безопасности / on-prem

Если данных мало — не домысливай.

---

## Статусы

Для каждого правила ставь:
- ✅ Выполнено
- ⚠️ Частично
- ❌ Не выполнено
- ⬜ Неприменимо
- ❓ Недостаточно данных

Не используй ❌ там, где просто не хватает информации.

---

## Для каждого правила фиксируй

1. Что есть
2. Чего не хватает
3. Почему это важно
4. Риск
5. Следующий шаг

## Логика аудита

### Блок A. Основа проекта

* правило 1: этапы
* правило 2: метрики
* правило 4: прототип
* правило 12: итеративность

### Блок B. Архитектура

* правило 3: рычаги улучшения
* правило 5: контекст / RAG
* правило 6: tools
* правило 7: workflow vs agent

### Блок C. Надежность и масштабирование

* правило 8: critic / HITL
* правило 9: размер модели
* правило 10: fine-tuning
* правило 11: serving optimizations

---

## Поведение skill

1. Не оценивай проект только по наличию технологий.
2. Не хвали агентность сама по себе.
3. Если нет метрик, тестовой выборки или прототипа — это почти всегда красный флаг.
4. Если пользователь просит короткое ревью — дай сначала summary, потом по желанию полный проход по 12 правилам.
5. Опирайся на `references/llm-checklist.md`.
