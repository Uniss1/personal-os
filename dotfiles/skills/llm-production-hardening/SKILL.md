# LLM Production Hardening

Используй этот skill, когда:
- прототип уже есть
- нужно улучшить надежность
- нужно снизить стоимость
- нужно ускорить систему
- нужно поставить цикл улучшений

---

## 1. Проверка ответов

Если по ответу можно понять, что система ошиблась —
строй critic.

### Вариант A. Reflection
Critic видит:
- задачу
- ответ
- план
- промежуточные шаги

Подходит, если:
- есть бюджет
- есть время
- нужны сильные reasoning-модели

### Вариант B. LLM-as-a-judge
Critic оценивает качество по критериям.

Подходит, если:
- нужно быстро
- нужен более дешевый контроль
- критерии качества хорошо описаны

### Если найден плохой ответ
1. Human-in-the-loop
2. LLM retry / пересдача с критикой

Если используешь LLM-цикл —
всегда ограничивай число итераций.

---

## 2. Размер модели

Проверь, что именно дает большая модель:
- знания
- рассуждения
- следование инструкции

Практика:
- знания часто компенсируются RAG
- рассуждения нужны не всегда
- для рутинных бизнес-задач часто подходят модели 1-10B

Перед тем как держаться за большую модель,
проверь, нельзя ли перейти на меньшую.

---

## 3. Fine-tuning

Используй fine-tuning, когда:
- малая модель плохо слушается промпта
- задачу нужно сфокусировать
- прототип уже понятен
- метрики можно пересчитать

Проверь:
- есть ли качественный датасет
- кто будет обучать
- как контролировать рост галлюцинаций
- как сравнить до/после на тестовой выборке

---

## 4. Serving-оптимизации

Проверь минимум:
- batching
- quantization
- response cache / KV-cache
- speculative decoding
- Flash Attention / Paged Attention
- фреймворк serving-а

При обсуждении всегда спрашивай:
- что важнее: throughput или latency
- on-prem или API provider
- контролирует ли команда serving stack

Если пользователь сидит только на внешнем API —
разделяй:
- оптимизации приложения
- оптимизации инференс-движка

---

## 5. Итеративный цикл

Всегда веди пользователя к циклу:

1. Собрать корзину входных задач
2. Посмотреть качество на корзине
3. Кластеризовать проблемы
   - по темам
   - по причинам
4. Для каждой проблемы:
   - гипотеза
   - реализация
   - метрика
   - принять / отклонить

### Источники корзины
- реальные запросы продукта
- опросы / интервью
- синтетические примеры

Главное правило:
не проверяй много гипотез одновременно.

---

## Поведение skill

1. Если пользователь хочет "сразу все оптимизации" — верни к порядку: качество -> размер модели -> tuning -> serving.
2. Если нет тестовой выборки — не обсуждай fine-tuning всерьез.
3. Если нет critic или HITL для критичных кейсов — подчеркни риск.
4. Если после запуска нет цикла оценки проблем — укажи это как системную слабость.
