# LLM Architecture Decisions

Используй этот skill, когда пользователь уже понимает задачу
и хочет выбрать правильный архитектурный рычаг.

## Главная развилка

Всегда различай:
- модели не хватает **знаний**
- модели не хватает **правильного поведения**
- модели нужно **выполнять действия**
- задачу нельзя надежно расписать жесткой логикой

### Соответствие решений
- знания -> RAG / контекст
- поведение -> prompt / workflow / fine-tuning
- действия -> tools
- плохая формализуемость -> agent

---

## Порядок выбора решений

Всегда начинай с самого дешевого и управляемого:
1. prompt
2. разбиение на подзадачи / workflow
3. RAG
4. tools
5. fine-tuning
6. agent

Не предлагай fine-tuning или agent как решение по умолчанию.

---

## 1. Prompt

Проверь сначала:
- точность формулировок
- few-shot
- reasoning, если он реально помогает
- structured output
- разбиение одной большой задачи на несколько шагов

Если задача плохо решается одним промптом —
предложи workflow.

---

## 2. RAG

Используй RAG, если модели не хватает знаний.

Важно:
RAG — это не только векторный поиск.
Допустимы:
- текстовый поиск
- поиск по структуре / БД
- векторный
- гибридный

Помоги выбрать:
- что именно ищем
- где хранятся знания
- как формируется поисковый запрос
- как найденный текст вставляется в контекст
- что делать, если поиск ничего не нашел

Не предлагай fine-tuning как основной способ добавления знаний.

---

## 3. Контекст

Считай контекст рабочей памятью LLM.

В контекст могут входить:
- факты о продукте
- правила
- история диалога
- профиль пользователя
- описание инструментов
- результаты tool-вызовов
- промежуточные действия системы
- критика от judge/critic модели

Правила:
- добавляй все нужное
- убирай шум
- следи за длиной окна
- если заранее неясно, что именно нужно модели — используй retrieval

---

## 4. Tools

Используй tools, когда требуется:
- точный расчет
- SQL / API / поиск
- операции вне текста
- детерминированная логика

Главный принцип:
всю детерминированную логику выноси из LLM в код.

LLM должна:
- понять, что надо сделать
- корректно вызвать инструмент
- использовать его результат

LLM не должна:
- имитировать калькулятор
- вручную сортировать вместо кода
- проверять строгие условия там, где есть алгоритм

### Structured Output
Если важна точность аргументов —
используй structured output.

Сначала:
- опиши инструмент в промпте
- укажи аргументы
- укажи формат результата

Только если этого не хватает —
рассматривай fine-tuning под tool use.

---

## 5. Workflow vs Agent

### По умолчанию — workflow
Вы сами задаете последовательность шагов,
а LLM используется только там, где нужна недетерминированная логика.

Плюсы workflow:
- предсказуемость
- легче улучшать
- можно мерить этапы по отдельности
- дешевле и быстрее

Минус:
при росте числа интентов/сценариев
логика начинает разрастаться.

### Агент
Используй агента только если:
- задачу трудно формализовать
- порядок действий заранее неизвестен
- результат можно проверить

Где агент особенно уместен:
- код с тестами
- research с проверкой человеком
- персональный ассистент с подтверждением рискованных действий

Если верификация результата неочевидна —
предпочитай workflow.

---

## 6. Fine-tuning

Предлагай fine-tuning, когда:
- промпт уже улучшали
- workflow/RAG/tools уже проверили
- модель все еще плохо следует инструкции
- модель нужно сфокусировать на одной задаче

Не предлагай fine-tuning как загрузку знаний.

Помни:
- чаще нужен на этапе разработки, а не прототипа
- чаще нужен для небольших моделей
- может усилить галлюцинации при плохом исполнении

---

## Поведение skill

1. Сначала определи, чего не хватает: знаний, поведения, действий или формализуемости.
2. Не советуй fine-tuning до prompt/RAG без сильной причины.
3. Не романтизируй агентность.
4. Всегда предлагай workflow раньше агента, если задача формализуема.
5. Для RAG отдельно проговаривай поиск, вставку контекста и fallback.
