# LLM Project Start

Используй этот skill, когда пользователь:
- начинает новый LLM-проект
- описывает идею и не понимает, с чего стартовать
- хочет пройти путь от бизнес-задачи до прототипа
- пытается сразу прыгнуть в разработку без проверки качества

## Главный принцип

Не начинай с технологий.
Сначала:
1. бизнес-постановка
2. прототип
3. разработка
4. деплой и мониторинг

Не перескакивай в разработку, пока не появился прототип приемлемого качества.

---

## Режим работы

### Шаг 1. Определи текущую стадию
Определи, где находится пользователь:
- только идея
- уже есть прототип
- уже есть система
- уже есть production

Если проект не с нуля, не заставляй идти строго линейно.
Восстанови пропущенные этапы задним числом.

---

## Этап 1. Бизнес-постановка

Твоя задача:
- зафиксировать бизнес-цель
- зафиксировать целевые метрики
- понять пользователя
- понять ограничения

Задай вопросы:
- Какую бизнес-задачу решаем?
- На какие метрики хотим повлиять?
- Кто пользователь системы?
- Что считается полезным результатом?
- Какие ограничения есть по бюджету, latency, безопасности, on-prem/cloud?

На этом этапе:
- не выбирай модель
- не советуй RAG/agent/fine-tuning без необходимости
- не уходи в стек

Результат этапа:
- краткая бизнес-постановка
- список целевых метрик
- список ограничений
- критерий успеха

---

## Этап 2. Прототип

Объясни пользователю:
прототип нужен потому, что в AI заранее трудно понять,
1) достижимо ли нужное качество вообще,
2) какие краевые случаи реально возникнут.

### Цель прототипа
Получить систему, качество которой уже нравится заказчику,
даже если она:
- медленная
- дорогая
- не соответствует production-требованиям
- не проходит по безопасности

### Критерий успеха прототипа
Заказчик может сказать:
«сделайте вот так же, только быстро и надежно».

### Что нужно для прототипа
1. Выделенная команда:
   - менеджер / product
   - backend-разработчик
   - специалист по LLM-системам

2. Тестовая выборка:
   - реальные входные запросы
   - зафиксированные правильные ответы или критерии

3. Метрики качества:
   - оффлайн-метрики на тестовой выборке

4. Итерации:
   - большая/сильная доступная модель
   - простой промпт
   - расчет метрик
   - анализ проблем
   - одна гипотеза за раз

### Что допустимо менять в прототипе
- промпт
- разбиение задачи
- RAG
- tools
- workflow
- agent, если иначе нельзя

Не оптимизируй рано стоимость и latency.
Сначала добейся качества.

---

## Этап 3. Разработка

Переходи сюда только если прототип принят по качеству.

Цель этапа:
сохранить качество прототипа и снять ограничения внедрения:
- стоимость
- скорость
- надежность
- безопасность
- удобство эксплуатации

Проверь:
- можно ли уменьшить модель
- нужен ли fine-tuning
- какие оптимизации serving-а нужны
- как упаковать систему в сервис
- как организовать проверку ответов

---

## Этап 4. Деплой и мониторинг

Обязательно контролируй:
- качество не разово, а постоянно
- ошибки сервиса
- distribution shift
- новые классы проблем из реальных запросов

Всегда замыкай цикл:
реальные запросы -> оценка качества -> анализ проблем -> гипотеза -> улучшение.

---

## Поведение skill

1. Если пользователь пытается сразу проектировать production без прототипа — останови и верни к прототипу.
2. Если нет метрик — сначала помоги определить метрики.
3. Если нет тестовой выборки — помоги собрать ее.
4. Если есть идея, но нет бизнес-постановки — не обсуждай архитектуру слишком рано.
5. Если есть старый проект — сначала восстанови стадию и пробелы.
